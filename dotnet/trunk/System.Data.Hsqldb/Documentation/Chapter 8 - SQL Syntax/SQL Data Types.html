SQL Type Mapping Interpretation
  • The SQL Type Declaration column lists the SQL standard type name (or name commonly used by RDMS), along will allowable create parameters.

  • The Length or Precision column indicates the maximum length or precision of the type that can be declared / stored. Where 1 to System.Int32.MaxValue is stated, it is simply the syntactically enforced theoretical limit.

    In practice, because the 1.8.0 engine implements in-line row data storage for all types and length specifications, the maximum length of a CHAR[ACTER], [LONG]VARCHAR or [[LONG]VAR]BINARY object that can be stored and retrieved with satisfactory performance is dictated by the amount of physical memory, drive speed and CPU power available.

    For instance, it has been demonstrated that values of a few (e.g. 1-4) megabytes may be used successfully in production databases. However, this range should be treated as an upper limit since frequent handling of such values represents serious memory stress and flirts with possibly unrecoverable out-of-memory exception conditions; although occasional retrieval of rows that occupy 512KB or less is not likely to unduely impact performance on modern hardware, the best policy at present is to isolate all LOB storage (say column values that occupy greater than 8KB) to one or more optional join tables composed of no more than an ID column and a LOB value column and from which rows are retrieved only in response to explicit requests for the LOB data.

    For very large values (greater than a few hundred KB), it is far more performant to store in the database only paths to files containing LOB data and simply stream such data directly to the client, based on lookup of file paths.

  • The System Type column lists the recommended/default C# and/or CLR type mapping for the SQL Type, followed by the internal (i.e. native) representation used by the engine (which is useful to know when implementing database triggers).

Notes

Observe that the recommended CLR mapping for SQL FLOAT is System.Double. Because of the potential confusion with float (System.Single), it is recommended that the SQL DOUBLE declaration is used instead of SQL FLOAT.

Also observe that VARCHAR_IGNORECASE is a special, case-insensitive variation of VARCHAR. Note that this type is not portable and should be avoided unless absolutely required.

Finally, observe that TIMESTAMP(precision) accepts only 0 or 6 (the default), where zero (0) denotes that the fractional seconds component (ffffff) is not stored.

When omitted, the engine interprets length, precision and scale values to be those listed in the Length or Precision column. When specified, length, precision and scale values are stored in the object's descriptor and reported in meta data, but are not enforced by default. However, you can issue the following command to cause the engine enforce declared length, precision and scale, as well as to pad CHAR[ACTER] values to length:

    SET PROPERTY "sql.enforce_strict_size" TRUE

Please note that setting this property to TRUE does not check existing values in the database, only values that are subsequently inserted or updated. Thus, if absolute consistency is required, this command should be issued immediately after database creation, before any values are added to the database.

CHAR, VARCHAR and LONGVARCHAR columns are compared and sorted using the UTF-16 Big Endian binary collation by default. The SET DATABASE COLLATION command allows this behavior to be modified.

Columns of the type OTHER or OBJECT contain the serialized form of an Object in binary format. In general, to insert or update such columns, one should use a command parameter, as this allows the driver to automate the transformation. For scripting purposes, however, it is nice to know that the SQL string literal representation is the base 16 (hexadecimal) encoding of the serialized form.
